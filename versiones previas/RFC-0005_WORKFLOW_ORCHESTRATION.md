# 1. Purpose

This RFC defines the Workflow Orchestration Engine of ATLAS.

Workflows coordinate long-running, asynchronous and cross-system business processes.

Business rules remain inside Domain Services.

The Workflow Engine coordinates execution.

---

# 2. Objectives

The Workflow Engine exists to:

- automate repetitive operations
- coordinate external integrations
- execute asynchronous processes
- schedule recurring tasks
- orchestrate notifications
- execute long-running business operations
- improve operational efficiency

---

# 3. Architectural Principles

Workflows must be:

- event-driven
- idempotent
- observable
- retryable
- resumable
- tenant-aware
- independently deployable

Business logic is never implemented inside workflows.

---

# 4. High-Level Architecture

```
Business Event

↓

Workflow Service

↓

Workflow Engine (n8n)

↓

External Systems

↓

Completion Event

↓

Domain Services
```

The workflow engine executes orchestration, not business decisions.

---

# 5. Workflow Lifecycle

```

↓

↓

↓

↓

Completion

↓

```

# 8. Workflow Definition

Each workflow defines:

- workflow_id
- version
- owner
- trigger
- inputs
- outputs
- retries
- timeout
- permissions

Workflow definitions are version-controlled.

---

# 9. Task Execution

A workflow is composed of tasks.

Example:

```
Reservation Confirmed

↓

Send WhatsApp

↓

Generate Check-in Guide

↓

Create Cleaning Task

↓

Notify Owner

↓

Publish Event
```

Tasks execute independently whenever possible.

---

# 10. Event Integration

Every workflow begins with an event and ends by emitting one or more events.

Example:

```
ReservationConfirmed

↓

Workflow

↓

GuestWelcomeSent

↓

CleaningScheduled

↓

OwnerNotified
```

Workflows integrate naturally with the Event Bus.

---

# 11. AI Integration

AI Agents may request workflow execution.

Example:

```
Owner

↓

"Prepare everything for tomorrow's guests."

↓

Operations Agent

↓

Workflow Service

↓

Workflow Engine
```

AI plans.

Workflows execute.

---

# 12. Retry Policy

Failures follow:

```
Failure

↓

Retry

↓

Exponential Backoff

↓

Maximum Attempts

↓

Escalation

↓

Dead Letter Queue (future)
```

Retries must be idempotent.

---

# 13. Timeout Policy

Every workflow defines:

- execution timeout
- task timeout
- retry timeout

Infinite execution is forbidden.

---

# 14. Idempotency

Repeated execution must produce identical business outcomes.

Duplicate events must not create:

- duplicated reservations
- duplicated notifications
- duplicated payments
- duplicated cleaning tasks

---

# 15. Security

Workflow execution requires:

- authenticated trigger
- authorized execution
- tenant isolation
- secure secret management
- audit logging

Secrets are injected at runtime.

---

# 16. Observability

Every execution records:

- execution_id
- workflow_id
- tenant_id
- trigger
- duration
- retries
- external calls
- produced events
- final status

Workflow health must be continuously monitored.

---

# 17. Scalability

Workflows execute asynchronously.

Future evolution supports:

- distributed workers
- queue mode
- workload partitioning
- horizontal scaling

Workflow execution must never block user interaction.

---

# 18. Failure Isolation

Workflow failures must not interrupt:

- reservations
- guest conversations
- pricing
- authentication
- AI reasoning

Operational continuity is prioritized.

---

# 19. Anti-Patterns

Forbidden:

- business rules inside workflows
- direct database mutations
- long synchronous operations
- shared mutable workflow state
- hardcoded secrets
- channel-specific workflow logic

---

# 20. Future Evolution

The Workflow Engine is designed to support:

- distributed orchestration
- workflow version migration
- human approval steps
- AI-generated workflows
- adaptive execution
- event replay
- MCP-triggered workflows

---

# 21. Success Criteria

The Workflow Engine is considered correctly implemented when:

- business logic remains outside workflows;
- every workflow is event-driven;
- retries are safe;
- execution is observable;
- failures are isolated;
- AI can safely trigger workflow execution.

---

# 22. Summary

The Workflow Orchestration Engine provides the operational automation layer of ATLAS.

By separating workflow coordination from business logic, the platform remains modular, scalable and AI-native while enabling reliable automation across hospitality operations, communications and third-party integrations.
