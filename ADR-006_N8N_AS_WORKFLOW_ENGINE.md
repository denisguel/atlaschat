# 1. Context

ATLAS requires an orchestration engine capable of coordinating:

- asynchronous business processes
- third-party integrations
- scheduled jobs
- notifications
- AI pipelines
- background tasks
- event-driven automations

These workflows should remain outside the core business services.

---

# 2. Decision

ATLAS adopts **n8n** as the primary workflow orchestration engine for the MVP and early production phases.

Business logic remains inside Domain Services.

n8n orchestrates processes but does not own business rules.

---

# 3. Decision Drivers

Primary motivations:

- open source
- self-hostable
- low operational cost
- visual workflow builder
- extensive integration ecosystem
- webhook support
- event-driven execution
- AI integration capabilities

---

# 4. Responsibilities

n8n is responsible for:

- workflow orchestration
- integration execution
- scheduled tasks
- webhook processing
- notifications
- long-running processes
- retries
- external API coordination

---

# 5. Responsibilities Outside n8n

n8n must NOT own:

- reservation logic
- pricing calculations
- authorization
- authentication
- tenant isolation
- AI orchestration
- business validation
- persistence ownership

Those responsibilities remain inside the ATLAS platform.

---

Generated by Domain Events.

```

↓

Create Cleaning Task

↓

Notify Owner
```

---

## Schedule Trigger

Examples:

- every morning
- every hour
- every week

---

## Manual Trigger

Started by:

- Owner
- Admin
- AI Agent

---

## Webhook Trigger

External systems initiate workflow execution.

# 9. Communication Model

n8n receives commands from:

- Workflow Service
- Event Bus
- Authorized APIs

n8n never calls Domain Services directly without explicit API contracts.

---

# 10. Error Handling

Workflow failures:

- retry automatically
- exponential backoff
- execution logging
- dead-letter queue (future)

Business operations must continue even if a workflow fails.

---

# 11. Idempotency

All workflows must support:

- duplicate event detection
- safe retries
- idempotent execution

Repeated triggers must not produce duplicated side effects.

---

# 12. Security

Workflow execution must enforce:

- tenant isolation
- signed webhooks
- authenticated API calls
- secret management
- execution permissions

Secrets are never embedded in workflow nodes.

---

# 13. Observability

Each execution records:

- workflow_id
- execution_id
- tenant_id
- trigger
- duration
- status
- retries
- produced events

---

# 15. Scalability

Workflows execute independently.

Future scaling:

- multiple n8n workers
- queue mode

- workload partitioning

---

# 16. Benefits

The decision provides:

- rapid development
- visual automation
- reusable workflows
- reduced engineering effort
- strong integration ecosystem
- asynchronous execution

---

# 17. Trade-Offs

Accepted limitations:

- workflow version management
- visual flow complexity
- operational monitoring

These are acceptable for the MVP and growth stages.

---

# 18. Alternatives Rejected

## Custom Workflow Engine

Rejected.

Reasons:

- excessive development effort
- unnecessary complexity
- slower MVP

---

## Temporal

Rejected for MVP.

Reasons:

- steeper learning curve
- higher operational complexity

Potential future evaluation.

---

## Apache Airflow

Rejected.

Reasons:

- data engineering focus
- unsuitable for conversational workflows
- heavyweight infrastructure

---

# 19. Future Evolution

Possible migration path:

MVP

↓

n8n

↓

Queue Mode

↓

Distributed Workers

↓

Dedicated Workflow Platform (if required)

The platform architecture should not require redesign.

---

# 20. Success Criteria

This ADR is considered successful when:

- workflows remain independent of business logic;
- integrations are orchestrated without tight coupling;
- failures do not impact core platform operations;
- workflows are observable and retryable;
- AI can trigger workflows through controlled interfaces.

---

# 21. Status

Accepted.

n8n becomes the official workflow orchestration engine for ATLAS.

It is responsible for coordinating asynchronous processes and external integrations while preserving the separation between AI reasoning, business logic and infrastructure.
